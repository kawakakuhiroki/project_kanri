# AGENTS.md

## 目的

- プロジェクト管理アプリ（project-kanri）の実装/保守作業の前提と運用ルールを共有する。

## 技術スタック

- Frontend: HTML / CSS / JavaScript（Vanilla）
- View: JSP
- Backend: Java 17 / Servlet（Jakarta Servlet 5.0）
- JSON: Jackson
- DB: MySQL（InnoDB）
- Data Access: JDBC（PreparedStatement必須）
- Build: Maven（war）
- Test: JUnit5

## 仕様優先順位

- 現行ソースコードを最優先
- 設計書（仕様の背景・画面/業務ルールの補足として参照）
- ソースコードと設計書に差異がある場合は、現行ソースコードを正とし、必要に応じて設計書を追随更新する
- 仕様未定は `TODO 仕様未定` を明記する

## 実装ルール（要点）

- ユーザーから「実装して」「変更して」「修正して」「OK」の明示指示がある場合のみ、ファイル変更・コード生成・変更系コマンド（例: `rm`, `mv`, `apply_patch`, `git commit`）を行う
- 調査・確認のための読み取り系コマンドは実行してよい
- 指示がない場合は調査・提案・方針提示に留め、いかなる変更も行わない
- 迷う場合は実装せず、確認を取る
- 変更前に「これから変更します」と宣言し、明示OKがない場合は中断する
- ソースを変更する場合、影響範囲に応じてテストコードを追加/更新する。該当なしの場合は理由を明記する
- ソースを変更した場合、mvn testを実行し、結果を報告する（mvn test は例外として明示OK不要）
- ソースを変更した場合、mvn packageを実行し、warを再生成する（mvn package は例外として明示OK不要）
- git commit を行う場合、コミットメッセージは日本語で記述し、変更の目的と主な修正内容が分かる具体的な表現にする
- 「修正」「対応」「変更」だけの曖昧なコミットメッセージは禁止とし、必要に応じて理由や背景を含める
- コミットメッセージは原則として件名と本文を使い分ける
  - 件名: 変更内容の要約を1行で記載する
  - 本文: 背景と主な修正点を記載し、必要に応じて理由や補足を加える
- アプリ仕様/業務ルール/画面・APIの詳細は現行ソースコードに従い、不足時は設計書（最新版）を参照する

## 設計書ルール

- ソースコードを追加/修正/削除するたびに docs/ 配下の最新版の外部設計書・詳細設計書を確認する。内容を変更する必要がある場合に限り、上書きせず新バージョンで最新化する
- 設計書（.xlsx）はExcelまたは表計算ライブラリ（例: openpyxl 等）で編集してよい。XML直接編集は禁止。
- 設計書の変更は必ず仕様本体に直接反映する。差分欄の追加は禁止。更新時は変更履歴シートに「日付/修正対象/変更内容」を記載し、修正対象はシート名のみ（接頭辞なし）、1修正対象=1行で記載する。
- 設計書の変更があった場合、変更箇所と理由を返答すること
- 設計書を修正する必要がなかった場合、その旨を返答すること
- 外部設計書は プロジェクト管理_外部設計書_vXX.xlsx 形式で採番し、docs/ に保存する。最新版は vXX が最大のものとし、旧版は削除しない。
- 詳細設計書は プロジェクト管理_詳細設計書_vXX.xlsx 形式で採番し、docs/ に保存する。最新版は vXX が最大のものとし、旧版は削除しない。

## コメント方針

- 入口コメント: クラス/メソッドの役割を1〜2行で明示（何を/なぜ）
- 処理の意図: 仕様起因・業務ルール起因の分岐には理由を書く
- データの流れ: 入力→検証→変換→保存→出力の区切りに短いコメント
- 境界条件: null/空/0.00/範囲チェックは守るルールを明記
- 副作用: DB更新・トランザクション・セッション操作は必ず明示
- 過剰は避ける: 代入や単純なifにはコメント不要
- 学習用の優先順位: 「なぜ/仕様理由/副作用/境界」を優先し、「何をしたか」の説明は最小限にする
- 削除対象の例: get/setの説明、doGet/doPost等の自動生成コメント、メソッド名の言い換え、重複コメント（// と /** */ の同内容）
- 追加対象の例: 仕様理由、業務ルール、例外理由、トランザクション/セッションなどの副作用、処理の段階区切り

## コーディング規約（学習用・統一整形）

- Java整形は **Google Java Style** に統一する
  - 例: import順は **アルファベット順（自動整形）**
    例: `jakarta.* → org.* → jp.co.*`（実際は全体で並ぶ）
  - Javadocは「コメント整形ルール」に従う（固定ラベル形式）
  - 1行に詰め込まず **1ステートメント1行**
- JS整形は **Prettier準拠**
  - インデント2スペース / セミコロンあり
  - 業務ロジックを含む関数はブロック・改行で記述する
  - 短いイベントハンドラやコールバックは1行記述を許可する
- CSS整形は **Prettier / stylelint**
  - インデント2スペース
  - **minify禁止、可読性優先**

### コメント整形ルール（学習用）

- ファイル冒頭に以下を統一記述する
  - `[役割] / [入力] / [出力] / [依存]`
- JavaのJavadocは以下に統一する
  - `[目的] / [入力] / [出力] / [影響]`
- `[影響]` は「戻り値以外への影響」を記述する
- getter/setter のJavadocは省略する（冗長化防止）
- “なぜ:” ラベルは使わず、自然文で理由を書く
- 重要画面は `[業務ルール]` を追加する
  - 対象: `project / projects / workflow / reports / admin_users`
- 対象外: `docs/`, `target/`, `.DS_Store`, `*.jar`

**フォーマットはこの規約に統一する。**
**保存時の自動整形（IDE/Prettier/Google Java Format）を推奨。**
